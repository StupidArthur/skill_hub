# 开发工程纪律

本技能以**开发流程**为主线，将工程纪律内嵌到每个阶段。调用时机：**开始一次开发或修改工作时一次调用**，之后按流程推进，不在 Vibe Coding 中途零散触发。

## 核心红线（贯穿全流程，每次对话生效）

以下规则优先级最高，影响每一个阶段：

- **谋定后动**：收到需求后禁止直接写代码，必须先走 [需求阶段](requirement.md) 和 [设计阶段](design.md)，只有用户明确同意后才能开始编码。
- **删代码确认**：覆盖或删除老代码前，必须向用户汇报并取得二次确认，确认后方可清理。
- **需求边界**：需求超出当前模块职责时，指出不合理之处并给出符合架构原则的替代方案。
- **按需执行**：严格按用户指令执行，不主动多做未要求的事（如额外写测试、额外验证、额外重构）。验证类操作前先征求同意。

## 开发流程

按以下五阶段顺序推进。每个阶段有明确的完成标志，未完成不进入下一阶段。

### ① 需求阶段

确认、拆分、明确需求边界。需求阶段不涉及技术选型。

- 读 [需求理解](requirement.md)
- 产出：需求文档（写法见 [文档与知识管理](doc-management.md)）
- 完成标志：用户确认"需求理解无误"

### ② 设计阶段

设计的第一步是技术选型，然后输出方案设计。

- 读 [方案设计](design.md)（含选型决策树，指向 [design/](design/README.md) 子目录的各技术栈规范）
- 设计阶段须检查横切规则：
  - 若涉及会丢数据的程序（批处理、长任务），须遵循 [运行安全与运维](runtime-safety.md)
  - 若涉及 LLM 调用的功能，须遵循 [LLM 集成功能设计](llm-integration.md)
- 产出：设计文档（写法见 [文档与知识管理](doc-management.md)）
- 完成标志：用户确认"设计方案同意"

### ③ 实现阶段

按设计编码。

- 读 [代码编写规范](coding-standards.md)
- 实现时按选型结论参考对应技术栈规范（[design/](design/README.md) 子目录）
- 实现中若涉及横切规则，落地 [运行安全与运维](runtime-safety.md) 和 [LLM 集成功能设计](llm-integration.md) 的设计决策
- 完成标志：编码完成，准备进入测试

### ④ 测试阶段

对纯业务逻辑执行测试。

- 读 [测试纪律](testing.md)
- 产出：测试记录（写法见 [文档与知识管理](doc-management.md)）
- 完成标志：核心业务逻辑测试通过

### ⑤ 提交与打包阶段

版本管理、提交、打包。

- 读 [版本管理](version-control.md) 和 [构建与打包](build-package.md)
- 产出：更新记录（随提交同步）
- 完成标志：提交完成，产物已打包

## 横切规则

以下规则不单独成阶段，被流程阶段引用：

| 规则 | 性质 | 被谁引用 |
|------|------|---------|
| [文档与知识管理](doc-management.md) | 产品规范 | ①②④（需求文档、设计文档、测试记录） |
| [运行安全与运维](runtime-safety.md) | 开发规范 | ②③（设计评估、实现落地） |
| [LLM 集成功能设计](llm-integration.md) | 开发规范 | ②③（设计评估、实现落地） |

## 适用范围

本技能是**语言与框架无关**的通用流程纪律，但设计阶段内嵌了技术选型（桌面 GUI / B/S 网站 / CLI 工具，Go / Python）。选型后的具体技术栈规范在 [design/](design/README.md) 子目录中。
