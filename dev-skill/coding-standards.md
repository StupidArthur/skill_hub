# 代码编写规范

> 流程阶段③。承接 [设计阶段](design.md) 的确认方案，按设计编码。以下规则为最高执行指令。本规范语言与框架无关。

## 架构与模块化

- **高内聚低耦合**：具备独立能力的代码必须封装为独立模块。
- **出入口管理**：
  - 对外服务（HTTP API 等）：路由和参数定义必须与 API 文档严格对应并同步更新。
  - 模块内部接口：必须在模块文件最上方用文档注释声明对外接口。**只有明确声明的接口才允许被外部调用，禁止跨模块调用内部私有函数。**
- **扁平化调用**：代码调用层次不要太深，每一层封装必须有明确意义，拒绝无谓的过度封装。
- **参数控制**：严格区分参数的归属模块，绝不让参数扩散到非必要的层级。
- **受控的整洁重构**：坚决抵制"同功能存在多版本冗余"的兼容性代码。但执行覆盖或删除老代码前，必须向用户汇报并取得二次确认。

## 变量与文件处理

- **消灭魔法数字**：涉及物理意义、反复用到的数字必须定义为常量。可能在运行中修改的配置项，必须置于模块的最上方。
- **文件命名法则**：
  - 代码中涉及的文件名必须是变量或参数，禁止写死字符串。
  - 输出文件名尽量由输入文件名推导（如输入 `data.csv`，输出 `data_processed.csv`）。
  - 多次运行生成的输出文件，**必须带上时间戳标签**。
- **数据格式**：优先使用标准化的数据格式（csv、json 或轻量级 DB）。

## 调试与注释

- **函数入口**：在 `main` 函数中，优先使用直接修改函数参数的方式进行调试，不要默认实现复杂的命令行解析（CLI）。
- **注释要求**：为每个核心模块、类、方法编写详尽的文档注释和行内注释，重点解释**"为什么这么写（业务背景/逻辑考量）"**而不仅仅是翻译"写了什么"。

## 前端通用

- 强调组件封装与复用，避免重复造轮子。
- UI 风格与所选组件库的官方规范保持一致，确保简洁专业。
- 具体前端技术栈规范（React + Shadcn 等）由 [dev-route-selection](../dev-route-selection/shared/frontend-components.md) 提供。
