# 需求理解

> 流程阶段①。本阶段产出需求文档，文档写法见 [文档与知识管理](doc-management.md)。

## 原子规则

- **收到新需求后绝对不要立刻开始写代码。**
- 必须先向用户输出对需求的理解，确认理解无误后才能进入设计阶段。
- 需求超出当前模块职责时，主动指出不合理之处，并给出符合架构原则的替代方案，而非机械执行。

## 需求理解输出

按以下结构输出对需求的理解，等用户确认：

```
1. 需求复述：用自己的话复述需求，确认理解无误
2. 边界识别：需求涉及哪些模块/不涉及哪些，明确职责边界
3. 影响面：会动到哪些现有功能
4. 需求合并空间（如有）：当用户连续提出多个层层推进的需求时，主动寻找"需求合并"的优化空间。例如用户先要求写 A→B，后又要求写 B→C，应主动提出"是否直接构建端到端 A→C 处理流"。
```

## 完成标志

用户确认"需求理解无误"后，本阶段结束，进入 [设计阶段](design.md)。需求文档同步更新。
