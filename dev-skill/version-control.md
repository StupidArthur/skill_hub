# 版本管理规范

> 流程阶段⑤。承接测试阶段，执行版本管理、提交与更新记录。

## 原子规则

- **绝对禁止直接执行 commit**：用户表示要 commit 时，禁止直接执行 `git commit`，必须先走变更分析流程。
- 编译缓存、依赖目录（如 `__pycache__`、`node_modules`、`.venv`、`build/`）必须在 `.gitignore` 中。
- 打包产物（如 `.spec`、`dist/`）绝对不允许被 Git 追踪。

## 多步流程

### 项目初始化检查

1. 启动或接到新任务时，隐式执行 `git status`。
2. 若非 Git 仓库，询问用户："当前目录尚未初始化 Git，是否需要我执行 `git init` 并生成 `.gitignore`？"

### 里程碑提醒

3. 阶段性任务完整完成后，**主动停下来提醒用户**："目前 [某某功能] 已更新完毕且测试通过，是否需要进行版本提交？"

### 提交前变更分析

4. 用户要求 commit 后，先执行 `git status` 查看变更文件列表。
5. 执行 `git diff` 和 `git diff --cached` 查看具体变动。
6. 向用户输出**变更影响分析报告**：
   - 影响模块：列出受影响的模块或业务域。
   - 具体变动：用自然语言总结新增、修改或删除了哪些核心逻辑。
7. 询问："以上是本次提交的影响面分析，是否同意进入提交环节？"

### 交互式提交

8. 用户同意后，根据变动推荐规范 commit message（格式 `<type>(<scope>): <subject>`）。
9. 询问："这是我推荐的 commit message，您确认直接使用，还是需要修改？"
10. 用户确认后执行 `git add .` 和 `git commit -m`。
11. 提交后告知当前最新 commit hash。

## 能力要求

- **版本演进总结**：用户询问历史版本时，用 `git log --oneline -n 10` 等命令，以清晰列表总结最近的版本演进。
- **版本差异解读**：用户询问版本差异时，用 `git diff <A> <B>` **解读**模块差异和核心逻辑变动，禁止只丢原始 diff 文本。
