# LLM 集成功能设计

> 横切规则（开发规范）。适用于开发涉及调用大模型（LLM）做下游决策的功能。在 [设计阶段](design.md) 评估是否适用，在实现阶段落地。

## 核心原则

涉及调用大模型的功能，必须遵循**"单一职责 + 规则化包围"**原则。禁止裸调大模型解决宽泛问题。

## 原子规则

- **功能单一**：每个 LLM 调用点只解决一个明确、狭窄的问题。一个功能若需多步推理，拆成多个单一调用点串联，而非一次调用解决全部。
- **规则化包围**：LLM 调用必须被一份结构化的 skill 或 prompt 约束，明确输入格式、输出格式、行为边界。禁止"把用户输入直接丢给大模型，让它自由发挥"。
- **禁止宽泛委托**：不向 LLM 委托宽泛、开放的问题（如"帮我分析这个系统"）。将宽泛问题拆解为若干个有明确输入输出的子任务，每个子任务单独调用。

## 反模式与正例

- ❌ **裸调**：`llm.chat(user_input)`，无 prompt 约束，输出格式不可控。
- ❌ **宽泛委托**：让一个 LLM 调用同时完成"理解需求 + 生成方案 + 写代码 + 测试"。
- ✅ **单一 + 包围**：`llm.chat(skill_prompt + structured_input)`，prompt 规定只做某一步，输出固定结构（如 JSON）。

## 设计模板

设计 LLM 功能时，先定义以下五项，再写代码：

1. **单一职责**：这次调用只解决什么狭窄问题（一句话能说清）。
2. **输入契约**：给 LLM 的输入是什么结构。
3. **输出契约**：要求 LLM 返回什么结构（优先 JSON schema 等可解析格式）。
4. **行为边界**：LLM 能做什么、不能做什么。
5. **失败兜底**：LLM 输出不符合契约时怎么处理（重试、降级、报错）。

## 与 skill 机制的关系

本规则所说的"规则化包围"，最佳实践就是用 skill 机制本身：把 LLM 的职责、输入输出契约、行为边界写成一份 skill（含 frontmatter 的 name/description + 正文约束），让 LLM 在 skill 框架内执行，而非自由发挥。
