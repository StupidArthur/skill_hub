# 目录级结构规范

> 本文件被 [SKILL.md](SKILL.md) 第三步、第四步引用。目录级 skill 的内容分类、路由写法、入口内容规范。

## 一、内容分类：总共要有几方面的内容？写在哪里？

目录级 skill 的内容分四类，各有归属：

| 内容类别 | 写在哪里 | 判据 | dev-skill 例子 |
|---------|---------|------|---------------|
| **入口内容** | SKILL.md | 贯穿全流程 / Agent 一加载就要知道的 | frontmatter、核心红线、流程主线、横切引用表 |
| **流程阶段文件** | 每阶段一个 .md | 属于某个特定阶段，有明确完成标志 | requirement.md、design.md、testing.md |
| **横切规则文件** | 独立 .md | 不属于某阶段，但多个阶段都要引用 | doc-management.md、runtime-safety.md |
| **子目录** | 阶段内容量大时建 | 单个 .md 装不下的主题性内容 | design/（选型内容，入口 README.md） |

### 关键原则

- **流程阶段文件和横切规则文件是平级的 .md**，都在 skill 根目录下。不要用目录区分它们——靠 SKILL.md 的流程主线和横切表区分归属。
- **子目录只在单个 .md 装不下时建**。比如 dev-skill 的 design 阶段有 16 个选型文件，才建 design/ 子目录；requirement 阶段内容少，一个 requirement.md 就够。
- **子目录入口用 README.md**，不用 SKILL.md。避免被 Agent 误认为独立 skill。

### 目录结构示例

```
skills/<skill-name>/
├── SKILL.md              入口（frontmatter + 红线 + 流程主线 + 横切表）
├── stage-1.md            流程阶段①
├── stage-2.md            流程阶段②
├── stage-3.md            流程阶段③
├── crosscut-a.md         横切规则 A
├── crosscut-b.md         横切规则 B
└── stage-2/              子目录（stage-2 内容多时）
    ├── README.md         子目录入口（不是 SKILL.md）
    ├── topic-1.md
    └── topic-2.md
```

## 二、路由写法：引导文件要怎么写？

**核心原则：不写独立的路由表，路由内嵌在流程节点里。**

### 流程节点的标准写法

每个流程阶段在 SKILL.md 中这样写：

```markdown
### ② 设计阶段

设计的第一步是技术选型，然后输出方案设计。

- 读 [方案设计](design.md)
- 若涉及会丢数据的程序，须遵循 [运行安全](runtime-safety.md)
- 产出：设计文档（写法见 [文档管理](doc-management.md)）
- 完成标志：用户确认"设计方案同意"
```

### 三条路由规则

1. **流程节点写"读哪个文件"，不写"什么时机读"**——时机就是流程位置本身。写"读 design.md"而不是"设计时读 design.md"。

2. **横切规则在阶段里直接引用，不写"若涉及 X 另读 Y"**——规则文件内部自己写明什么情况怎么办。正确写法是"须遵循 [运行安全](runtime-safety.md)"，规则文件内部再写"适用于中途退出会丢数据的程序"。

3. **子目录入口用 README.md 引用**——写"见 [design/](design/README.md)"，不写"见 design/SKILL.md"。

### 反面教材

以下写法是错误的（纯路由表式入口）：

```markdown
## 路由表

| 触发条件 | 读哪个文件 |
|---------|-----------|
| 写单文档时 | single-doc.md |
| 写目录级时 | directory.md |
```

为什么错：这是纯路由表，没有流程主线。Agent 不会在每个任务节点重新扫描路由表做状态匹配，路由表只是"躺在上下文里的文字"，可靠性有限。正确做法是把引用内嵌在流程节点里。

## 三、入口内容：哪些东西写在 SKILL.md 上？

入口 SKILL.md 必须有四块，顺序固定：

### 1. frontmatter

```yaml
---
name: <skill-name>
description: <做什么 + 何时触发 + 兜底句>
---
```

详见 [frontmatter 规范](frontmatter.md)。

### 2. 核心红线（内联）

贯穿全流程、违反后果不可逆的规则。内联在入口，因为：
- Agent 一加载就要知道，不能等读子文件
- 红线入选判据：违反后果不可逆 + 每次都可能触发 + 单条 ≤ 2 行

示例（dev-skill）：
```markdown
## 核心红线

- **谋定后动**：收到需求后禁止直接写代码，必须先走需求阶段和设计阶段。
- **删代码确认**：覆盖或删除老代码前，必须向用户汇报并取得二次确认。
```

### 3. 流程主线

阶段列表，每阶段写四要素：

| 要素 | 说明 | 必填 |
|------|------|------|
| 做什么 | 一句话说明本阶段职责 | 是 |
| 读哪个文件 | 引用阶段文件/横切规则 | 是 |
| 产出 | 本阶段产出什么 | 视情况 |
| 完成标志 | 什么状态算完成，可进入下一阶段 | 是 |

示例（dev-skill 的设计阶段）：
```markdown
### ② 设计阶段

设计的第一步是技术选型，然后输出方案设计。

- 读 [方案设计](design.md)（含选型决策树，指向 [design/](design/README.md) 子目录）
- 若涉及会丢数据的程序，须遵循 [运行安全](runtime-safety.md)
- 产出：设计文档
- 完成标志：用户确认"设计方案同意"
```

### 4. 横切规则引用表（如有）

用表格说明哪些横切规则被哪些阶段引用：

```markdown
## 横切规则

| 规则 | 性质 | 被谁引用 |
|------|------|---------|
| [文档管理](doc-management.md) | 产品规范 | ①②④ |
| [运行安全](runtime-safety.md) | 开发规范 | ②③ |
```

### 入口不能有的内容

- 各阶段的具体规则步骤 → 外置到阶段文件
- 详细技术规范 → 外置到子目录
- 9 维中的 Skills/Workflows 细节 → 外置
- 大段代码示例 → 外置到子文件或 examples/

**判据**：这条内容是否"Agent 一加载就要知道"？是 → 内联；否 → 外置。

## 四、轻量/重量维度判据

9 维在目录级 skill 中的归属，不按维度名字分，按三属性分：

| 属性 | 内联（入口） | 外置（子文件） |
|------|------------|------------|
| 触发频率 | 每次都可能触发 | 特定阶段才触发 |
| 后果 | 违反后不可逆 | 违反后可修正 |
| 篇幅 | 单条 ≤ 2 行 | 需要步骤/示例/代码 |

这个判据能解释所有归属：
- Role/Profile/Background/Goals：高频 + 短 → 内联（入口 frontmatter 或开头）
- 核心红线：高频 + 不可逆 + 短 → 内联
- Skills/Workflows 细节：特定阶段 + 长 → 外置
- Constraints 详细规则：特定场景 + 长 → 外置（核心部分可内联为红线）

## 五、子文件规范

### 流程阶段文件

- 无 frontmatter（只有入口 SKILL.md 有）
- 开头用 blockquote 标注所属阶段：`> 流程阶段②。承接 [需求理解](requirement.md)`
- 结尾写完成标志：`用户确认"XXX"后，本阶段结束，进入 [下一阶段](next.md)`

### 横切规则文件

- 无 frontmatter
- 开头用 blockquote 标注性质和被谁引用：`> 横切规则（开发规范）。被 [设计](design.md) 和 [实现](coding-standards.md) 引用`
- 内部写明"什么情况适用"——因为入口只引用文件名，适用判断在文件内部

### 子目录

- 入口用 README.md，不用 SKILL.md
- README.md 无 frontmatter，是普通索引文件
- 子目录内的文件用相对路径互引（如 `../shared/frontend-stack.md`）
