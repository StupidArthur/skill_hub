# Skill 编写规范

本技能指导如何编写 skill。**所有 skill 的入口都必须是流程型**——SKILL.md 不能是纯路由表，必须有"先做什么、再做什么"的流程主线。如果内容没有流程，那它的流程就是"只有一个环节"，也要写出来。

## 通用红线

- **入口必须流程型**：SKILL.md 正文必须是流程主线，不能是纯路由表。流程节点内嵌文件引用，不单独建路由表。
- **单文档也要在目录下**：即使是单文档 skill，也要放在 `skills/<name>/SKILL.md`，不能是裸 .md 文件。
- **子目录入口用 README.md**：目录级 skill 的子目录入口用 README.md，不用 SKILL.md，避免被误认为独立 skill。
- **frontmatter 必备**：SKILL.md 必须有 frontmatter（name + description），这是 Agent 发现和加载 skill 的唯一依据。
- **description 含兜底句**：description 不能只枚举触发场景，必须有一句兜底（如"任何涉及 X 的工作都应调用"），防止枚举遗漏导致 skill 不被加载。

## 编写流程

按以下四步顺序推进。每步有明确的产出，未完成不进入下一步。

### 第一步：评估内容，选形态

判断 skill 内容适合单文档还是目录级。

**判据**：内容能在一个 SKILL.md 里讲清楚且不臃肿 → 单文档；内容跨多个独立主题、单文件装不下 → 目录级。

| 形态 | 适用场景 | 结构 |
|------|---------|------|
| 单文档 | 内容少、9 维能覆盖 | `skills/<name>/SKILL.md`（含 9 维） |
| 目录级 | 内容多、有多个主题 | `skills/<name>/SKILL.md`（入口）+ 子文件 |

- 单文档规范见 [单文档 9 维规范](single-doc.md)
- 目录级规范见 [目录级结构规范](directory.md)

**完成标志**：确定形态，向用户说明理由并确认。

### 第二步：写 frontmatter

无论单文档还是目录级，SKILL.md 都要有 frontmatter。

- 格式规范见 [frontmatter 规范](frontmatter.md)
- 关键：description 必须写清"做什么 + 何时触发 + 兜底句"

**完成标志**：frontmatter 写好，name 和 description 都符合规范。

### 第三步：写入口 SKILL.md

入口 SKILL.md 必须是流程型，包含四块（顺序固定）：

1. **frontmatter**（第二步已写）
2. **核心红线**（内联）——贯穿全流程、违反后果不可逆的规则
3. **流程主线**——阶段列表，每阶段写：做什么 + 读哪个文件 + 完成标志
4. **横切规则引用表**（如有）——哪些规则被哪些阶段引用

入口不能有的：各阶段具体规则步骤、详细技术规范、9 维中的 Skills/Workflows 细节——这些外置。

- 目录级的入口写法详见 [目录级结构规范](directory.md)
- 单文档的入口就是 SKILL.md 本身，9 维中的 Workflows 即流程主线

**完成标志**：入口 SKILL.md 写好，流程主线清晰，每阶段有文件引用和完成标志。

### 第四步：写子文件（目录级才有）

单文档跳过此步。目录级 skill 按内容类别写子文件：

| 内容类别 | 写在哪里 | 判据 |
|---------|---------|------|
| 流程阶段文件 | 每阶段一个 .md | 属于某个特定阶段，有明确完成标志 |
| 横切规则文件 | 独立 .md | 不属于某阶段，但多个阶段都要引用 |
| 子目录 | 阶段内容量大时建 | 单个 .md 装不下的主题性内容 |

- 详细规范见 [目录级结构规范](directory.md)

**完成标志**：所有子文件写好，被入口流程节点正确引用。
