## 一、分类与编号说明

| 前缀 | 类型 | 含义 | 来源 |
|------|------|------|------|
| **R** | 缺陷修复 | 评审已验证的工程缺陷 / 自洽性问题 / 维护负担 | review_merged.md |

**两类条目的本质区别**：
- R 类是"修坏的东西"——模板通不过校验器、路径不可移植、校验器缺失等，不修会影响开箱即用。
- M 类是"加新的约束"——给生成流程加 outline 评审暂停点、规则冲突必须问用户。不加也能跑，但少了人机协同门。

## 三、两类条目的交叉关系与张力

> 落地前必读：M 类（需求）与 R 类（缺陷）并非完全独立，存在以下交互。

### 3.1 协同关系（M 类依赖 R 类先修好）

| 需求 | 依赖的缺陷 | 说明 |
|------|-----------|------|
| **M1** Phase 2 机械翻译 | **R1** 模板过校验器 / **R5** 风格 A 校验器 | M1 的 Phase 2 是"verbatim 翻译，0 思考"，前提是模板与校验器自洽。若 R1/R5 未修，机械翻译会产出通不过校验的 HTML，Phase 2 反复返工。**建议 R1/R5 先于 M1 落地。** |
| **M2** 规则冲突问用户 | **R6** 同名类冲突 | R6 本身就是一条"规则冲突"隐患（同名类语义不同）。M2 的冲突处理框架正好能兜住 R6 未根治前的误用。两者可并行，M2 是 R6 的过渡兜底。 |
| **M4** outline schema | **R4** 示例项目 | M4 的 outline 示例可直接作为 R4 示例项目的"规划层产物"，一份产出两用。 |

### 3.2 张力关系（M 类与 R 类方向冲突，需取舍）

| 冲突点 | M 类立场 | R 类立场 | 建议 |
|--------|---------|---------|------|
| **是否直接 patch SKILL.md** | M3 + personal_mod Q3 选了"A · 直接 patch（侵入式）" | R3/R8 强调上游同步、可移植、文档单一权威；硬编码路径问题正是"本地侵入式修改"的教训 | ⚠️ **建议改用 personal_mod 的 Q3 选项 B（轻量）或 C（不改 SKILL.md，靠补充规则文件）**。若必须 patch，强制配合 M3 的 `local-mods` metadata + git 版本控制，避免 `npx skills update` 覆盖丢失。 |
| **M1 暂停点 vs 生成效率** | M1 强制 BLOCKING，不批准不动 HTML | 三方评审未提效率问题，但 DS 提到"降低使用门槛" | M1 会增加提问次数，有经验用户"无脑生成"变慢。这是设计意图，可接受，但应在文档明示"首次/重要 deck 用 Phase 1，熟练后可跳过"。 |

### 3.3 落地位置差异（重要）

| 类别 | 落地目标 | 说明 |
|------|---------|------|
| R1–R18 | 当前仓库 `g:\skills\guizang-ppt-skill\` | 缺陷修复应回到上游仓库，通过 issue/PR 推动作者修复，惠及所有用户 |
| M1–M4 | personal_mod 指向本地安装副本 `C:\Users\Administrator\.agents\skills\guizang-ppt-skill\` | 需求增强是个人工作流偏好，不宜直接侵入上游；建议作为本地补充规则文件，或向上游提 feature request 讨论 |

> **结论**：R 类走上游修复路径，M 类走本地扩展路径。两者物理上不在同一份文件，但逻辑上构成同一份"改进计划"。

## 四、建议执行顺序

> 原则：先修自洽性（P0），再补质量兜底（P1 缺陷），最后加工作流增强（P1 需求）。

| 阶段 | 条目 | 产出 | 验收 |
|------|------|------|------|
| **阶段 0 · 立即** | R1 + R2 | 模板通过自家校验器；生成物本地 motion fallback 生效 | `node scripts/validate-swiss-deck.mjs assets/template-swiss.html` exit=0；断网打开生成 deck 动效正常 |
| **阶段 1 · 短期** | R3 + R4 + R5 + R7 | 路径可移植；有示例项目；风格 A 有校验器；data-theme 表述统一 | 新用户按文档能独立跑通；两套风格都有校验器 |
| **阶段 2 · 中期（需求增强）** | M1 + M2 + M4 | outline-driven 工作流 + 规则冲突处理 + schema | 跑 1 个 demo 验证 Phase 1/2 分界与冲突提问生效 |
| **阶段 3 · 中期（缺陷收尾）** | R6 + R8 + R9 + R10 + R12 | 类名前缀化；文档去重；模板可维护；CDN 降级明确；场景边界清晰 | 长期维护成本下降 |
| **阶段 4 · 长期** | M3 + R11 + R13–R18 | 本地修改可追踪；其余小问题清理 | `npx skills update` 后能从 metadata 识别本地改动 |
