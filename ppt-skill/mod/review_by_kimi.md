## 一、总体评价

**guizang-ppt-skill 是一份定位清晰、完成度很高的 Agent Skill。** 它不是简单地把 Markdown 转成 HTML，而是围绕"Agent 生成可直接演示的横向翻页网页 PPT"这个核心场景，提供了一整套从需求澄清、模板选择、内容填充、配图生成到质量校验的完整工作流。

在 Agent Skill 这个类别里，它属于**第一梯队**：文档齐全、模板可运行、有明确的美学系统、有自动化校验、有实战沉淀。适合用来快速产出演讲型 deck、发布会 slides、公众号/小红书封面等视觉物料。

---

## 二、核心亮点

### 1. 双风格系统覆盖完整

Skill 提供两条成熟的设计路线：

- **风格 A · 电子杂志 × 电子墨水**：衬线标题、WebGL 流体背景、暖色，适合人文、叙事、观点表达。
- **风格 B · 瑞士国际主义**：无衬线、网格点阵、单一高饱和锚点色，适合产品、数据、方法论表达。

两条路线不是换肤，而是各自独立的模板、布局、主题色和配图规则。这种"二选一"的收敛策略，既保证了输出质量，也降低了 Agent 的决策负担。

### 2. 工作流详细且可执行

[`SKILL.md`](./SKILL.md) 是一份写给 Agent 的操作手册：

- 有 7 问需求澄清清单，强制在动手前对齐风格、受众、时长、素材、图片需求、主题色、硬约束。
- 从拷贝模板、替换占位符、选择主题色，到填充内容、配图、自检、交付，每一步都有明确指令。
- 对常见返工点有预防性说明，例如"类名必须在模板 `<style>` 里有定义"、"不要接受用户自定义 hex 值"。

### 3. 瑞士风版式锁定 + 自动化校验

这是本 Skill 最大的亮点之一。

- [`references/swiss-layout-lock.md`](./references/swiss-layout-lock.md) 把瑞士风正文页严格限定在 22 个登记版式（`S01-S22`）内。
- 每个 `<section>` 必须声明 `data-layout`。
- 顶部标题默认左上对齐，SVG 不写可见文字，图片必须绑定 `data-image-slot`。
- [`scripts/validate-swiss-deck.mjs`](./scripts/validate-swiss-deck.mjs) 把上述规则自动化，会拦截未登记版式、P23/P24 实验结构、SVG 内文字、图片未绑定槽位等问题。

"规则文档 + 自动化校验"的组合在 Agent Skill 里比较少见，能显著提升生成质量的稳定性。

### 4. 模板可直接运行

两个模板都是单文件 HTML：

- 内置横向翻页（键盘、滚轮、触屏、底部圆点、ESC 索引）。
- 内置 WebGL 背景动画，且支持按 `B` 切换低性能静态模式。
- 内置 Motion One 入场动效，做了本地 + CDN 双保险。
- 不需要构建工具，浏览器直接打开即可演示。

### 5. 配图与封面能力完整

[`references/image-prompts.md`](./references/image-prompts.md) 提供了针对风格 A 和风格 B 的配图提示词库，覆盖纪实照片、信息图、流程图、对比图、系统关系图、截图再设计、数据大字报等类型。并规定了图片命名规范、落位比例和多平台封面规格（公众号 21:9、1:1 分享卡、小红书 3:4、视频号 16:9）。

## 四、优点

1. **实战沉淀感强**：从需求澄清、叙事弧、主题节奏到图片命名，很多细节都来自真实分享场景的迭代。
2. **美学系统自洽**：两套风格都有明确的设计原则、主题色、字体搭配和排版规则，不是拼贴式 CSS。
3. **Agent 友好**：大量规则被写成"Agent 能听懂的话"，例如"不要发明新类名"、"先选版式再生成图片"。
4. **质量兜底到位**：checklist、版式锁定、校验脚本三层防护，能拦住大部分低质量输出。
5. **扩展性合理**：配图流程可以自然延伸到多平台封面，让同一套视觉系统复用。

## 五、改进建议

### 1. 增加示例项目（高优先级）

当前仓库没有 `examples/` 目录。对于新用户来说，直接阅读 `SKILL.md` 和 `references/` 的学习成本较高。建议增加 1-2 个完整生成的示例 deck（例如一个 10 页杂志风、一个 12 页瑞士风），展示从模板到成品的完整结构。

### 2. 为风格 A 增加校验器（高优先级）

目前只有瑞士风有 [`scripts/validate-swiss-deck.mjs`](./scripts/validate-swiss-deck.mjs)。风格 A 同样有很多约束（主题色替换、`h-hero`/`stat-card` 等类名正确使用、主题节奏规则），建议增加 `validate-magazine-deck.mjs`，检查：

- 是否还有 `[必填]` 占位符未替换。
- `slide` 的 `class` 是否正确包含 `light`/`dark`/`hero light`/`hero dark`。
- 是否连续出现 3 页以上同主题。
- 使用的类名是否都在模板 `<style>` 中定义。

### 3. 补充国际化支持

模板默认中文，README 虽有英文版，但模板占位文案、类名说明、错误提示仍以中文为主。如果目标用户包含英文场景，可以考虑：

- 提供英文版模板占位文案。
- 在 `SKILL.md` 中增加"英文 deck"的处理说明。

### 4. 降低模板对外部 CDN 的强依赖

当前模板依赖 Google Fonts、Lucide 图标、Motion One 等 CDN。离线环境或网络不稳定时可能加载失败。可以考虑：

- 在仓库中提供本地字体/图标/Motion 的 fallback。
- 或在 README 中明确说明网络依赖。

### 5. 优化同名类的视觉差异风险

风格 A 和风格 B 中都有 `.h-hero`、`.lead`、`.meta-row` 等同名类，但视觉表现完全不同。虽然 Skill 文档反复强调不能混用，但这仍然是 Agent 容易踩的坑。可以考虑：

- 在风格 B 中使用前缀化的类名，例如 `.sw-h-hero`、`.sw-lead`，从命名层面避免混淆。
- 或在校验脚本中增加对"风格 A 类名出现在风格 B 模板中"的检测。

### 6. 拆分超长模板文件

[`assets/template.html`](./assets/template.html) 和 [`assets/template-swiss.html`](./assets/template-swiss.html) 把 CSS、WebGL shader、JS、HTML 结构全部放在一个文件里。虽然符合"单文件交付"的目标，但对于维护者来说阅读成本较高。可以考虑：

- 在开发时用多文件组织，发布/使用时再打包成单文件。
- 或在文件内用更清晰的 region 注释分隔各部分。

### 7. 明确不支持的场景边界

`README.md` 和 `SKILL.md` 已经列出了不适合的场景（大段表格、培训课件、多人协作编辑）。建议进一步强化：

- 增加"如果用户要求动画复杂交互/嵌入视频/实时数据"的拒绝或降级策略。
- 在需求澄清阶段增加一个问题："是否需要打印/导出 PDF？" 因为单文件 HTML 打印时的分页表现可能不稳定。

## 六、适合与不适合

### 适合

- 线下分享、发布会、demo day 的演讲 deck。
- 把长文章/方法论快速转成视觉化 PPT。
- 需要统一视觉风格的多平台封面（公众号、小红书、视频号）。
- 已经在用 Claude Code / Codex 且愿意接受"先对齐需求再动手"工作流的用户。

### 不适合

- 需要多人协作编辑的商务场景。
- 信息密度极高的培训课件或数据报表。
- 期望"一句话就出一整套完美 PPT"的懒人用户。
- 对离线运行、PDF 导出、视频嵌入有强需求的场景。

## 七、结论

**guizang-ppt-skill 是一份经过实战打磨、值得参考和复用的高质量 Agent Skill。** 它在"网页 PPT"这个垂类里建立了很高的标杆：不仅有模板，还有完整的工作流、美学规则、配图流程和质量校验。

如果维护者能补上示例项目、风格 A 校验器和更清晰的国际化/离线支持，它完全可以成为 Agent 生成 PPT 的首选 Skill 之一。
