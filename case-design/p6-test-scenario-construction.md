# P6 测试场景构建（Test Scenario Construction）

> 流程阶段⑥。承接 P5 的测试设计策略矩阵，把离散的测试点按业务时序串联成可执行的场景蓝图，供 P7 机械翻译为标准测试用例。

## 核心使命

把"碎片"组装成"故事"。

P5 输出的是独立的原子测试点（TP-001、TP-002…），每个测试点只回答"测什么"。P6 要做的是把这些原子按**真实用户的操作时序**串成一条条**有头有尾的业务叙事**，并为每个步骤绑定对应的测试点和代表性数据。P6 不写具体怎么点按钮/调接口，那是 P7 的事。

## 环节性质：一次性（One-shot）

| 维度 | 说明 |
|------|------|
| **模式** | **一次性组装编排**，非循环迭代 |
| **原因** | 输入（P5 策略矩阵）已是稳定、完整的决策产物。P6 只做"串联+数据赋值" |
| **错误处理** | 发现测试点之间有时序冲突或状态矛盾 → **回溯 P4**，重新审视测试点粒度或分类 |

## 输入标准（不满足不启动）

P6 启动前，以下输入必须就绪：

| 输入来源 | 要求 |
|---------|------|
| P5 策略矩阵 | 所有测试点已绑定方法+覆盖准则+优先级+轻量验证状态 |
| P3 领域模型 | 包含对象关系、状态机定义、属性数据类型及值域 |
| 前置依赖 | P5 已完成且无 High 优先级 Fail（P5 负责阻塞，P6 不重复检查） |

## P6 核心操作：如何把测试点串联成场景

### Step 1：识别"业务目标"（Aggregate Root + Goal）

从 P3 模型中读取聚合根（如 Order）及其核心 Goal（如"用户提交订单"）。一个 Goal 对应一组场景集合。

### Step 2：读取状态机，生成 Happy Path 骨架

从 P3 状态机中读取合法主路径（如 Draft → Pending → Paid → Shipped），这是 Happy Path 的**骨架**。

### Step 3：遍历 P5 策略矩阵，挂载测试点到步骤上

对每个测试点，判断它落在状态机路径的哪个步骤上，挂载上去。

| P5 测试点示例 | 落在哪个步骤 | 挂载方式 |
|-------------|------------|---------|
| TP-001：有效订单提交成功 | 步骤"提交→支付" | 挂载到"提交订单"步骤，作为该步骤的验证点 |
| TP-003：金额=0 被拒 | 步骤"提交→校验金额" | 不在 Happy Path 上 → 归入 Exception Path |
| TP-015：Draft→Pending 转换 | 步骤"提交" | 挂载到"提交订单"步骤 |

### Step 4：按"覆盖准则"生成场景分支

P5 为每个测试点标注了覆盖准则（如"覆盖所有合法+非法跃迁"）。P6 按此拆解场景数量：

| 覆盖准则 | P6 场景拆解动作 |
|---------|---------------|
| "覆盖所有合法跃迁" | 状态机有 N 条合法边 → 在 Happy/Alternate 路径中分别覆盖 |
| "覆盖所有非法跃迁" | 状态机有 M 条非法边 → 生成 M 个 Exception Path 场景 |
| "覆盖所有条件组合" | 决策表有 K 个组合 → 每个组合至少生成 1 个场景（可合并到不同步骤中） |
| "覆盖边界值开区间/闭区间" | min / max / min-1 / max+1 → 分别归入 Happy 或 Exception 场景 |

### Step 5：赋值"代表性数据"

从 P3 模型的属性定义域中读取数据类型和值域，为每个场景选用代表性值：

| P3 属性定义 | P6 选值策略 |
|-----------|-----------|
| 整数，范围 1-100 | Happy：选 50（典型值）；Exception：选 0 和 101 |
| 枚举，`{DRAFT, PENDING, PAID}` | Happy：选 DRAFT → PENDING → PAID 路径 |
| 字符串，非空，长度≤50 | Happy：选 "正常商品名"；Exception：选 空字符串 和 51 字符 |

> **关键原则**：P6 只从 P3 已定义的范围中选值，不发明新的数据类别。如果 P3 未定义值域，则 P6 **阻塞并回溯 P3 补充**。

## 步骤序列编写规范（P6 vs P7 的边界）

以下规范确保 P6 不越界进入 P7 的领域：

| ✅ P6 应该写的（业务叙事） | ❌ P6 不应该写的（UI/API 细节） |
|--------------------------|------------------------------|
| "用户创建订单" | "点击'新建订单'按钮，填写表单" |
| "用户提交订单" | "点击'提交'按钮" |
| "系统调用支付网关" | "调用 POST /api/pay 接口" |
| "系统扣减库存" | "UPDATE inventory SET qty = qty - 1" |
| "系统拒绝并提示错误" | "断言返回 code=400，message='金额无效'" |

> P7 负责把左侧的"业务叙事"翻译成右侧的"执行脚本"。P6 不关心实现方式（Web/App/API），只关心业务流程。

| 输入源 | P6 用不用 | 用途 |

## 场景编写示例（订单提交功能）

### P5 输入的测试点（部分）

| ID | 描述 | 方法 | 覆盖准则 |
|----|------|------|---------|
| TP-001 | 有效订单提交成功 | 用例测试 | Happy Path |
| TP-003 | 金额=0 时提交被拒 | ECP+BVA | 边界值：无效等价类 |
| TP-015 | Draft→Pending 状态转换 | 状态转换 | 合法跃迁 |
| TP-022 | 支付网关超时时订单回滚 | 错误猜测 | 恢复路径 |

### P6 产出的场景蓝图

**SC-001（Happy Path）：有效订单提交并支付成功**
- **前置**：用户已登录，购物车有商品
- **步骤 1**：用户创建订单，金额填入 **100**（有效典型值） → 关联 TP-001
- **步骤 2**：用户提交订单 → 关联 TP-015（Draft→Pending）
- **步骤 3**：系统调用支付网关，支付成功
- **预期**：订单状态=Pending，提示"提交成功"
- **覆盖准则达成**：覆盖 Happy Path 主线 + 有效等价类典型值 + 状态跃迁 Draft→Pending

**SC-003（Exception Path）：金额为 0 时提交被拒**
- **前置**：用户已登录，购物车有商品
- **步骤 1**：用户创建订单，金额填入 **0**（无效边界值） → 关联 TP-003
- **步骤 2**：用户提交订单
- **预期**：系统拒绝提交，提示"金额必须大于 0"，订单保持 Draft → 关联 TP-015（非法跃迁：验证状态未变）
- **覆盖准则达成**：覆盖边界值"无效等价类" + 非法状态跃迁检查

**SC-004（Recovery Path）：支付网关超时，订单回滚**
- **前置**：用户已登录，购物车有商品
- **步骤 1**：用户创建并提交有效订单 → 关联 TP-001、TP-015
- **步骤 2**：系统调用支付网关，**超时无响应** → 关联 TP-022
- **预期**：事务回滚，订单状态恢复为 Draft，提示"支付服务异常，请稍后重试"
- **覆盖准则达成**：覆盖恢复路径 + 事务一致性

## 完成标志与下一阶段

- **完成标志**：所有测试点均已归属到至少 1 个场景；五类场景覆盖矩阵填写完毕（无空白格）。
- **存在未归属测试点**：若有测试点未能串联进任何场景 → 检查是该测试点孤立（如纯算法验证不需要场景），若是则标注"N/A-无需场景"；若非孤立则 **回溯 P4 重新审视测试点粒度**。
- **进入下一阶段**：场景蓝图集合 + 覆盖矩阵确认无误后，进入 [测试用例生成](p7-test-case-generation.md)。
