# P3 需求对象化（Requirement Object Modeling）

> 流程阶段③。承接 [P2 需求解构](p2-requirement-analysis.md) 的结构化元素清单，一次性横向聚合成统一领域对象，供 [P4 测试点提取](p4-test-point-extraction.md) 使用。

## 核心使命

把 P2 打碎的元素卡片（8 类业务元素 + 归属标签），横向聚合（Modeling）成统一的领域对象（Requirement Object）。

P3 是**一次性建模（One-shot Modeling）**，不是循环迭代。它不做"补充需求"，只做"重组与抽象"——把散装的业务规则归类到领域实体和值对象，识别聚合根，梳理关系。P3 是连接"需求文本"和"测试分析"的**语义桥梁**。

## 与 P1、P2 的对比

| 维度 | P1 | P2 | P3 |
|------|----|----|----|
| 核心动作 | 完善（完善模糊点） | 解构（纵向切分） | 建模（横向聚合） |
| 模式 | 循环（Loop） | 一次性（One-shot） | 一次性（One-shot） |
| 核心思想 | 需求工程 | 规格说明分析（ISTQB 29119） | 领域驱动设计 + UML |
| 操作对象 | 自然语言文本 | 8 类元素卡片 | 元素卡片 → 领域对象 |
| 产出物 | 可测试需求（L3） | 结构化元素字典 | 统一领域模型（Requirement Object） |
| 错误处理 | 内部迭代消化 | 发现缺口→回溯 P1 | 发现无法聚合→回溯 P2 |

## 思想与标准支撑

### 核心思想：DDD 战术建模 + 面向对象分析（OOA）

P2 的 8 类元素是"散装的业务规则"——属性/条件/动作的堆砌，没有"归属感"。P3 要做的是把这些散装元素归类到领域实体（Entity）和值对象（Value Object），识别聚合根（Aggregate Root），梳理关系。

最终产出的 Requirement Object，本质是一个**轻量级领域模型（Domain Model）**——不含代码实现，但包含测试设计所需的所有业务语义上下文。

### 产物要素的工程学来源

| P3 产物要素 | 工程学来源 / 标准 | 在测试设计中的用途 |
|------------|------------------|-------------------|
| 对象（Object） | UML 类图的"类" / DDD 的"实体（Entity）" | 定义测试的操作主体（PID 控制器、用户账户、订单） |
| 属性（Attribute） | UML 类图的"属性" / DDD 的"值对象（Value Object）" | 等价类划分和边界值分析的作用域（温度值、转速值） |
| 方法/操作（Method） | UML 类图的"操作" / DDD 的"领域行为" | 对应 P2 的 Goal，测试用例要触发的主动作（启动调优、提交订单） |
| 关系（Relationship） | UML 的关联/聚合/组合 / DDD 的聚合根 | 集成测试和场景构建的上下游依赖 |
| 状态（State） | UML 状态机图 / DDD 的"状态模式" | 状态转换测试的蓝图（待调优→调优中→已完成） |
| 不变量（Invariant） | DDD 的"业务不变规则" | 对应 P2 的 Constraint，判定表/决策表的核心条件（温度必须在 0-100°C） |

## 关键操作：从"二维表格"到"立体模型"

- **P2 的输出**是 8 类元素 + 归属标签的独立列表（Actor 列表、Input 列表、Output 列表…），相当于一张扁平表格，每行是离散元素。
- **P3 的操作**：
  - 识别哪个 Input 属于哪个 Object（`温度值`属于`温度传感器对象`，`目标转速`属于`电机对象`）
  - 识别 Object 之间的生命周期依赖（聚合关系）
  - 识别 Object 内部的状态跃迁（哪些操作改变哪些属性）

这一步做完，P4 才能从"这个 Input 要测什么边界值"这种低级问题，上升到"这个 Object 在什么状态下被调用、会触发什么行为"这种高级测试分析。

## 输入标准（必须满足，否则 P3 不启动）

- P2 输出的**结构化需求元素清单**（8 类元素齐全，无留空，`N/A` 有明确标注）
- **额外要求**：P2 输出的元素之间**必须带"归属标签"**——即 P2 拆解时已标注"这个 Input 属于哪个业务概念"。若 P2 没带这个标签，P3 无法建模，必须回溯 P2 补充

## 输出标准

一份 **Requirement Object 模型文档**，包含五部分：

| 部分 | 内容 |
|------|------|
| 对象清单 | 所有识别出的领域对象及其类型（实体 / 值对象 / 枚举） |
| 属性清单 | 每个对象的属性，带数据类型和业务约束 |
| 操作清单 | 每个对象对外提供的业务方法，带输入参数和返回值 |
| 关系描述 | 对象之间的关联、聚合、依赖关系 |
| 状态机描述 | 核心对象的状态定义及合法跃迁路径 |

这份模型作为 **P4 的唯一下游输入**，P4 不再查阅 P2 的原始列表或 P1 的 PRD。

## 层间防火墙

P3 发现 P2 输出的 9 类元素无法聚合成完整对象（如 Input 和 Output 之间缺少逻辑归属，无法归类到任何实体）时：

- **不允许在 P3 内部猜测或修补**
- **不允许自行补充业务规则**
- 必须**立即停止并回溯到 P2**——P2 的元素提取不完整或粒度不对，P3 无法建模

P3 只负责如实"组装"，确保每层只做一件事，职责单一。

## 完成标志

Requirement Object 模型文档（五部分齐全）生成，作为 P4 唯一输入，本阶段结束，进入 [测试点提取](p4-test-point-extraction.md)。
