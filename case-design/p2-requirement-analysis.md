# P2 需求解构（Requirement Analysis）

> 流程阶段②。承接 [P1 需求澄清与评审](p1-requirement-review.md) 输出的成熟度 L3 需求，一次性拆解为结构化元素清单，供 [P3 需求对象化](p3-requirement-object-modeling.md) 建模。

## 核心使命

把自然语言需求机械地、无歧义地拆解成结构化元素。

P2 是**一次性转换（One-shot Translation）**，不是循环迭代。输入是已经过 P1 验证、达到 L3（Testable）的稳定需求，P2 不做"完善"或"补充"，只做"格式转换"——把人类友好的自然语言，降维成机器/后续流程友好的结构化元素卡片。

## 与 P1 的本质区别

| 维度 | P1 | P2 |
|------|----|----|
| 输入 | 模糊的、不确定的自然语言 | 已达 L3 的稳定需求 |
| 模式 | 循环迭代（Loop） | 一次性转换（One-shot） |
| 职责 | 完善需求（定性） | 解构需求（定量拆解） |
| 错误处理 | 内部迭代消化 | 发现缺口→回溯 P1，不擅自修补 |

## 思想与标准支撑

P2 不是凭空发明，背后有成熟的工程思想和标准：

- **ISTQB 核心方法论**：基于规格说明的测试设计（Specification-Based Testing）前置准备。P2 把需求文本转化为"测试条件原材料"——即 8 类元素。
- **ISO/IEC/IEEE 29119**：该标准明确要求测试设计前必须完成"需求元素识别（Requirement Element Identification）"。P2 的 8 类元素严格对应 29119 中"测试依据分析（Test Basis Analysis）"必须提取的维度。

### 8 类元素的工程学来源

| 元素 | 工程学来源 | 测试设计中的用途 |
|------|-----------|---------------|
| Actor | UML 用例图中的参与者 | 定义测试执行者 / 角色权限变体 |
| Goal | 用例的"主成功场景" | 定义测试的终极验证目标（验收标准） |
| Input | 形式化方法中的"前置条件"输入域 | 等价类 / 边界值分析的原料 |
| Output | 形式化方法中的"后置条件"输出域 | 预期结果验证的原料 |
| Constraint | 业务规则 / 不变式（Invariant） | 判定表 / 决策表的条件项 |
| State | 有限状态机（FSM） | 状态转换测试的原料 |
| Exception | 健壮性测试（Robustness Testing） | 异常路径 / 负面测试的原料 |
| Dependency | 架构依赖 / 外部接口 | 集成测试顺序 / 桩模块设计的依据 |

> 一言以蔽之：P2 把自然语言需求"降维打击"成 8 个独立维度的结构化卡片，确保后续 P3 对象化时，不会因为"看漏了一个输入参数"而导致整个测试点提取崩塌。

## 输入标准（必须满足，否则 P2 不启动）

- P1 输出的需求澄清报告 + 最终版需求文档
- 需求成熟度达 L3，Quality Gate G1-G5 全绿
- **特别强调**：必须确认"无 Open Critical Missing / Conflict"。Exception 项哪怕写着"确认不存在"，也得有明确记录，不能是空白

## 输出标准

- 一张**结构化需求元素清单**：每条需求对应一个独立的元素卡片
- 每张卡片**至少**填满 9 类槽位（8 类业务元素 + 归属标签，有值或明确标注 `N/A`，**不允许留空**）
- 输出物直接作为 P3 的唯一下游输入，P3 不再回溯查阅原始 PRD

### 元素卡片结构

| 槽位 | 说明 | 填写要求 |
|------|------|---------|
| Actor | 参与者 | 有值或 N/A |
| Goal | 业务目标 | 有值或 N/A |
| Input | 输入 | 有值或 N/A |
| Output | 输出 | 有值或 N/A |
| Constraint | 约束 | 有值或 N/A |
| State | 状态 | 有值或 N/A。须包含：合法状态列表、合法跃迁及触发条件、已知非法跃迁（供 P3 建状态机） |
| Exception | 异常 | 有值或"确认不存在"，禁止留空 |
| Dependency | 依赖 | 有值或 N/A |
| **归属** | 该需求元素所属的业务概念/领域对象 | 必填，为 P3 建模提供聚合依据 |

> **归属标签**是 P2 输出给 P3 的关键衔接字段：P3 靠它判断哪些元素属于同一个领域对象。若 P2 未标注归属，P3 无法建模，会回溯 P2 补充。

### 示例

| 槽位 | 内容 |
|------|------|
| Actor | 操作员 |
| Goal | 执行 PID 调优 |
| Input | PID 参数 |
| Output | 调优结果 |
| Constraint | 管理员权限 |
| State | 待调优→调优中→完成 |
| Exception | PLC 离线 |
| Dependency | PLC 在线 |
| 归属 | PID 调优任务 |

## 层间防火墙

P2 发现遗漏（如解构时发现缺少 Exception）时：

- **不允许在 P2 内部循环补齐**
- **不允许猜测或自行补充**
- 必须**立即停止并回溯到 P1**，让 P1 再跑一轮补充需求

这保证了层间防火墙是干净的——P2 只做"如实拆解"，不做"完善需求"。P1 负责需求的定性完善，P2 负责需求的定量拆解，职责不交叉。

## 完成标志

每条需求都输出一张填满 9 类槽位（8 类业务元素 + 归属标签）的元素卡片（无留空），本阶段结束，进入 [需求对象化](p3-requirement-object-modeling.md)。
