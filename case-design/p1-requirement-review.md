# P1 需求评审（Requirement Clarification）

> 流程阶段①。承接原始需求（SRS / PRD / 用户故事），输出需求澄清报告供 P2 解构。

## 核心使命

发现需求缺陷，并转换为可执行的需求澄清问题。

本阶段不是"审计模式"（只列问题就结束），而是"澄清模式"——测试团队真正创造价值的地方，不是指出需求有问题，而是把模糊需求推动成可实现、可验证、可测试的需求。这样后面的 P2、P3、P4 才有稳定输入，否则所有测试设计工作都建立在猜测上。

## 两种模式对比

| 模式 | 输出 | 价值 |
|------|------|------|
| 审计模式（Audit Mode） | 问题列表 | 有限——测试人员看完"然后呢？" |
| 澄清模式（Clarification Mode） | 澄清问题 | 直接推动需求补全 |

本阶段采用**澄清模式**。

## 输入

| 类型 | 示例 |
|------|------|
| PRD | 产品需求文档 |
| SRS | 软件需求规格说明书 |
| Standard | IEEE、国标 |
| Design | 设计文档 |
| Story | 用户故事 |

## 输出：需求澄清报告（四层）

P1 的核心产物是**需求澄清报告（Requirement Clarification Report）**，而非问题列表。报告含四层，逐层递进：

| 输出层 | 内容 | 面向对象 |
|--------|------|---------|
| Layer 1 · 缺陷 | Requirement Defects | 测试专家 |
| Layer 2 · 影响 | Impact Analysis | 测试负责人 |
| Layer 3 · 澄清问题 | Clarification Questions | 产品经理 / 需求方 |
| Layer 4 · 建议更新 | Suggested Requirement Update | 文档作者 |

> Layer 3（澄清问题）比前两层都重要——它是推动需求收敛的直接动力。

### 示例

| 项目 | 内容 |
|------|------|
| Defect | 未定义权限 |
| Impact | 无法设计权限测试 |
| Questions | 哪些角色可执行？权限不足如何处理？ |
| Suggested Update | 增加角色权限矩阵章节 |

## 评审维度与引导提问

六个评审维度，每个维度发现缺陷后**必须**生成对应的澄清问题，不能只报缺陷不提问。

### Completeness（完整性）

需求"用户可以启动 PID 调优"，发现 Input/Output/Constraint/Exception 缺失，不要只输出"缺少 Input"，而应生成：

| 类别 | 澄清问题 |
|------|---------|
| Input | 启动 PID 调优时需要输入哪些参数？ |
| Input | 参数是否允许为空？ |
| Output | 调优成功后输出哪些结果？ |
| Constraint | 是否要求管理员权限？ |
| Exception | PLC 离线时如何处理？ |

### Consistency（一致性）

发现章节 3"允许删除"与章节 7"禁止删除"冲突，不要只说"存在冲突"，而应输出：

| 类型 | 澄清问题 |
|------|---------|
| Conflict | 删除功能是否最终支持？ |
| Conflict | 哪个章节描述为最终版本？ |
| Conflict | 是否存在条件删除场景？ |

### Testability（可测试性）

需求"系统响应迅速"不可测试，不要只说"不可测试"就结束，而应问：

| 类型 | 澄清问题 |
|------|---------|
| Testability | 响应时间指标是多少？ |
| Testability | 是否有 SLA 要求？ |
| Testability | 统计口径是什么？ |
| Testability | 在什么负载条件下验证？ |

### 其余维度

| 维度 | 英文 | 检查内容 | 引导提问方向 |
|------|------|---------|------------|
| 正确性 | Correctness | 是否符合业务 | 业务规则应以哪份文档为准？ |
| 歧义性 | Ambiguity | 是否有模糊表述 | 该表述具体指什么场景？ |
| 可追踪性 | Traceability | 是否可关联来源 | 该需求来源是哪个用户故事 / 标准？ |

## 完成标志

需求澄清报告（四层）输出，澄清问题已提交需求方，用户确认"需求已澄清可进入解构"后，本阶段结束，进入 [需求解构](p2-requirement-analysis.md)。
