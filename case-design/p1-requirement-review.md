# P1 需求澄清与评审（Requirement Clarification & Review）

> 流程阶段①。承接原始需求（SRS / PRD / 用户故事），多轮迭代将需求推进到可测试状态，输出需求澄清报告供 P2 解构。

## 核心使命

将需求推进到可测试状态。

P1 不是一次性的 Review，而是**需求澄清循环（Requirement Clarification Loop）**——每轮发现最重要的问题、推动需求继续完善，不追求一次评完。现实中需求评审根本不可能一轮结束，P1 的工作模式就是多轮迭代。

本阶段不是"审计模式"（只列问题就结束），而是"澄清模式"——测试团队真正创造价值的地方，不是指出需求有问题，而是把模糊需求推动成可实现、可验证、可测试的需求。这样后面的 P2、P3、P4 才有稳定输入，否则所有测试设计工作都建立在猜测上。

## 主要用户

| 用户 | 最关心的三个问题 |
|------|---------------|
| 测试分析师 | 还有哪些需求没说清？（Completeness） |
| 测试负责人 | 哪些地方说得矛盾？（Consistency） |
| 领域专家 | 哪些地方我没法设计测试？（Testability） |

P1 不输出几十页评审意见，而优先输出 **Top N 阻碍测试设计的问题**。

## 输入

| 类型 | 示例 |
|------|------|
| PRD | 产品需求文档 |
| SRS | 软件需求规格说明书 |
| Standard | IEEE、国标 |
| Design | 设计文档 |
| Story | 用户故事 |

## 工作模式：多轮迭代

```
Version 1 → Review → 生成澄清问题 → 测试专家找产品确认 → 补充需求
Version 2 → Review → 再次生成问题 → ... 持续迭代
Version 3 → Review → ...
```

每轮的目标不是"发现所有问题"，而是"本轮发现最重要的问题，推动需求继续完善"。

例如：
- 第一轮发现：未说明权限、未说明输入参数
- 产品补充后，第二轮发现：参数范围未定义、权限异常流程未定义
- 产品补充后，第三轮发现：状态转换规则冲突

这才符合真实需求评审。

## 输出：需求澄清报告

P1 的核心产物是**需求澄清报告（Requirement Clarification Report）**。报告含以下部分：

### 1. 问题列表（带状态）

每个问题带状态字段，支持跨轮追踪：

| 字段 | 含义 |
|------|------|
| ID | 问题编号（RR-001） |
| Status | Open / Clarified / Resolved |
| Round | 第几轮发现 |
| Priority | 优先级 |
| 类型 | Completeness / Consistency / Testability / ... |
| 描述 | 缺陷描述 |
| 影响 | 对测试设计的影响 |
| 澄清问题 | 向需求方提问 |
| 建议更新 | 建议的需求修改 |

示例：

| ID | 状态 | 轮次 | 类型 | 描述 | 澄清问题 |
|----|------|------|------|------|---------|
| RR-001 | Resolved | Round1 | Completeness | 未说明权限 | 哪些角色可执行？权限不足如何处理？ |
| RR-002 | Clarified | Round2 | Completeness | 参数范围未定义 | PID 参数允许范围是多少？ |
| RR-003 | Open | Round3 | Consistency | 状态转换规则冲突 | 调优中能否被中断？ |

### 2. Delta Analysis（变更分析）

多轮迭代必须输出变更分析，否则第三轮开始测试专家不知道哪些是新发现、哪些已解决：

| Round | 新增 | 已解决 | 仍存在 |
|-------|------|--------|--------|
| R1 | 15 | 0 | 15 |
| R2 | 6 | 10 | 11 |
| R3 | 2 | 9 | 4 |

### 3. 需求成熟度评估

每轮输出当前成熟度等级 + 距下一级缺什么。

## 需求成熟度模型

成熟度是**状态机**，不是分数——百分比指标（如"完成度 90%"）看起来科学但无法客观计算：简单登录 4/4 项 = 100%，复杂 PID 系统 8/9 项 = 89%，后者显然更完整却分数更低。所以用状态机。

| 等级 | 名称 | 判定标准 |
|------|------|---------|
| L0 | Draft | 原始需求，未经评审 |
| L1 | Reviewed | 完成第一轮评审 |
| L2 | Structured | 核心需求元素已补齐 |
| L3 | Testable | 已满足测试设计输入条件（Quality Gate 全通过） |
| L4 | Ready | 已进入测试分析阶段（P2） |

> **L3 = 允许进入 P2**。这是状态机的关键跃迁点。

每轮输出示例：

```
当前等级：L2
距 L3 还缺：
- 参数范围定义
- 权限矩阵定义
- 异常流程定义
```

这让 P1 从"挑错工具"变成"需求成熟度推进工具"。

## Quality Gate（质量门禁）

不叫 Exit Criteria，叫 Quality Gate——用"通过/未通过"的必须项，不用无法客观计算的百分比。

| Gate | 说明 |
|------|------|
| G1 | 无 Critical Missing |
| G2 | 无 Critical Conflict |
| G3 | 无 Untestable Requirement |
| G4 | 核心需求元素已明确（见下表） |
| G5 | 测试专家确认可进入测试分析 |

### G4 核心需求元素明细

| 必须项 | 要求 |
|--------|------|
| Goal | 已明确 |
| Actor | 已明确 |
| Input | 已明确 |
| Output | 已明确 |
| Constraint | 已明确或确认不存在 |
| Exception | 已明确或确认不存在 |
| Open Critical Missing | 0 |
| Open Critical Conflict | 0 |

### 判定

- 全部 Gate 通过 → **P1 PASS**，成熟度达到 L3，进入 [需求解构](p2-requirement-analysis.md)
- 任一未通过 → **继续澄清**，进入下一轮迭代

## 评审维度与引导提问

六个评审维度，每个维度发现缺陷后**必须**生成对应的澄清问题，不能只报缺陷不提问。

### Completeness（完整性）

需求"用户可以启动 PID 调优"，发现 Input/Output/Constraint/Exception 缺失，不要只输出"缺少 Input"，而应生成：

| 类别 | 澄清问题 |
|------|---------|
| Input | 启动 PID 调优时需要输入哪些参数？ |
| Input | 参数是否允许为空？ |
| Output | 调优成功后输出哪些结果？ |
| Constraint | 是否要求管理员权限？ |
| Exception | PLC 离线时如何处理？ |

### Consistency（一致性）

发现章节 3"允许删除"与章节 7"禁止删除"冲突，不要只说"存在冲突"，而应输出：

| 类型 | 澄清问题 |
|------|---------|
| Conflict | 删除功能是否最终支持？ |
| Conflict | 哪个章节描述为最终版本？ |
| Conflict | 是否存在条件删除场景？ |

### Testability（可测试性）

需求"系统响应迅速"不可测试，不要只说"不可测试"就结束，而应问：

| 类型 | 澄清问题 |
|------|---------|
| Testability | 响应时间指标是多少？ |
| Testability | 是否有 SLA 要求？ |
| Testability | 统计口径是什么？ |
| Testability | 在什么负载条件下验证？ |

### 其余维度

| 维度 | 英文 | 检查内容 | 引导提问方向 |
|------|------|---------|------------|
| 正确性 | Correctness | 是否符合业务 | 业务规则应以哪份文档为准？ |
| 歧义性 | Ambiguity | 是否有模糊表述 | 该表述具体指什么场景？ |
| 可追踪性 | Traceability | 是否可关联来源 | 该需求来源是哪个用户故事 / 标准？ |

## 完成标志

Quality Gate（G1-G5）全部通过，需求成熟度达到 L3（Testable），本阶段结束，进入 [需求解构](p2-requirement-analysis.md)。
