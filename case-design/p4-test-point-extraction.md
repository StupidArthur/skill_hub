# P4 测试点提取（Test Point Extraction）

> 流程阶段④。承接 [P3 需求对象化](p3-requirement-object-modeling.md) 的 Requirement Object 模型，一次性穷举扫描出所有测试关注点，供 [P5 测试设计映射](p5-test-design-mapping.md) 使用。

## 核心使命

对 P3 的领域模型进行地毯式扫描，穷举出所有值得测试的关注点。

P4 是**一次性穷举扫描**，不是循环迭代。它不做"完善模型"，也不做"理解语义"，而是对模型的每一个缝隙（属性、操作、状态、关系、约束）问一句："这个点，要不要测？"——它是一张"检查清单引擎"。

## 与 P1-P3 的对比

| 维度 | P1 | P2 | P3 | P4 |
|------|----|----|----|----|
| 核心动作 | 完善（定性） | 解构（纵向切分） | 建模（横向聚合） | 扫描（穷举枚举） |
| 模式 | 循环（Loop） | 一次性（One-shot） | 一次性（One-shot） | 一次性（One-shot） |
| 核心思想 | 需求工程 | 规格说明分析（29119） | 领域驱动设计 + UML | 基于模型的测试分析（ISTQB 测试条件识别） |
| 操作对象 | 自然语言文本 | 文本 → 8 类元素 | 元素 → 领域对象 | 领域对象 → 测试关注点 |
| 产出物 | 可测试需求（L3） | 结构化元素字典 | 统一领域模型 | 九大分类测试点集合 + 覆盖矩阵 |
| 错误处理 | 内部迭代 | 缺口→回溯 P1 | 无法聚合→回溯 P2 | 模型不完整→回溯 P3 |

## 思想与标准支撑

### 核心思想：基于模型的测试分析 + ISTQB 测试条件识别

在 ISTQB 体系中，测试分析师拿到测试依据（Test Basis）后，第一件事就是**识别测试条件（Test Conditions）**——即"什么东西需要被测试"。

P3 输出的 Requirement Object，本质是形式化程度极高的测试依据。P4 要做的，是对该模型执行一次**结构化遍历（Structured Walkthrough）**，针对模型中每一个构造元素，按预设分类框架提取测试关注点。

这与编译器对 AST（抽象语法树）做"语义分析"的逻辑类似——遍历每一个节点，检查并生成中间表示。P4 就是测试领域的"语义分析器"。

## 九大测试点分类

九大分类严格对齐 **ISO/IEC 25010 产品质量模型**与 **ISTQB 通用测试类型**，并结合 P1-P3 的元素体系做的映射：

| # | 测试点分类 | 对应 P3 模型要素 | 对应 ISO 25010 特性 | 提取逻辑 |
|---|-----------|---------------|-------------------|---------|
| 1 | 功能正确性（Functional Correctness） | 对象的操作（Method）+ 输出（Output） | 功能完整性/正确性 | 给定输入，输出是否与规约一致？ |
| 2 | 业务规则与逻辑（Business Rules） | 不变量（Invariant）+ 约束（Constraint） | 功能正确性 | 业务规则（如：折扣不能叠加）是否被正确执行？ |
| 3 | 输入边界与等价类（Boundary & Equivalence） | 属性（Attribute）的数据类型 + 值域 | 功能恰当性 | 每个输入参数的边界值、无效等价类是什么？ |
| 4 | 状态转换（State Transition） | 状态机（State Machine）定义 | 功能正确性/可靠性 | 所有合法跃迁是否可达？非法跃迁是否被阻止？ |
| 5 | 异常与健壮性（Exception & Robustness） | 异常（Exception）定义 | 容错性/健壮性 | 依赖故障、无效输入、资源耗尽时，系统如何表现？ |
| 6 | 交互与集成（Interaction & Integration） | 对象间关系（Relationship/Dependency） | 互操作性 | A 调用 B 时，数据传递是否正确？时序是否合规？ |
| 7 | 权限与安全（Permission & Security） | Actor + 操作（Method） | 安全性 | 无权限角色是否无法执行敏感操作？ |
| 8 | 非功能性（Performance/Resource） | 约束中的性能指标（响应时间/吞吐量） | 性能效率 | 在特定负载下，操作耗时是否满足 SLA？ |
| 9 | 数据完整性（Data Integrity） | 属性 + 对象关系（持久化/事务） | 可靠性/数据一致性 | 事务回滚后数据是否保持一致？并发写入是否冲突？ |

> **原则**：不是每个需求都覆盖全部 9 类。P4 的做法是**遍历所有分类，针对模型要素匹配命中——命中的记录，未命中的标注 N/A 并说明原因**。这确保不是凭直觉漏测，而是刻意确认过"确实不存在"。

## 提取操作：如何从一个对象上"榨出"测试点

以 P3 的一个"订单对象（Order）"为例：

| P3 模型要素 | P4 提取出的测试点（示例） | 归属分类 |
|-----------|------------------------|---------|
| 属性：`totalAmount`（Decimal，≥0） | 1) totalAmount = 0 时是否处理？<br>2) totalAmount = 最大值+1 时是否拒绝？ | #3 边界与等价类 |
| 操作：`submit()`（需 `status = Draft`） | 1) Draft 状态下提交 → 成功转为 Pending<br>2) Pending 状态下提交 → 报错 | #4 状态转换 |
| 不变量：`discount ≤ 30%` | 1) 折扣=30% 通过<br>2) 折扣=31% 被拒绝 | #2 业务规则 |
| 关系：依赖 `PaymentGateway` | 1) PaymentGateway 超时 → 订单状态不变<br>2) PaymentGateway 返回失败 → 订单标记为 Failed | #5 异常与健壮性、#6 交互 |

## 输入标准（必须满足，否则 P4 不启动）

- P3 输出的 **Requirement Object 模型文档**必须完整包含：对象清单、属性（含类型/约束）、操作（含参数/返回值/前置条件）、关系、状态机
- **硬性门禁**：模型的"属性数据类型"和"状态机完整跃迁路径"不得为占位符或模糊描述。若发现 `temperature: 数值类型` 这种过于宽泛的描述，P4 无法做等价类划分，须回溯 P3 精化

## 输出标准

一张 **测试点清单（Test Point List）**，每条格式：

| 字段 | 内容 |
|------|------|
| TP-ID | TP-001 |
| 描述 | 具体到"测什么" |
| 归属分类 | #1-#9 |
| 来源要素 | 回溯到 P3 的哪个对象/属性/操作 |
| 优先级 | High / Medium / Low（测试专家判定） |

**必须附带「九大分类覆盖矩阵」**——横轴是 9 个分类，纵轴是 P3 识别出的所有对象，单元格打勾或标注 N/A 原因。这张矩阵是 P4 完成的**有形证据**，证明没有盲区。

## 层间防火墙

P4 扫描模型时发现某个属性没有数据类型、或某个状态跃迁缺少触发条件（即模型本身不完整）时：

- **不允许猜测数据类型或补齐状态跃迁**
- P4 只能在完整模型上做"提取"，不能在残缺模型上做"想象"
- 必须**立即停止并回溯到 P3**

这确保测试点集合的可信度——每个测试点都能明确追溯到模型的哪个具体要素。

## 完成标志

测试点清单 + 九大分类覆盖矩阵输出（命中的记录、未命中的标注 N/A 原因），本阶段结束，进入 [测试设计映射](p5-test-design-mapping.md)。
