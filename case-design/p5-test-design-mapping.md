# P5 测试设计映射（Test Design Mapping）

> 流程阶段⑤。承接 P4 的测试点集合，为每个测试点匹配最合适的 ISTQB 测试设计方法，输出带覆盖准则的策略矩阵，供 P6 场景构建使用。

## 核心使命

为每个测试点绑定一个"武器"。

P5 不是创造新测试点，也不是设计具体用例，而是**决策引擎**——它读取 P4 的测试点清单（已分类、已标优先级），对照标准方法库查表匹配，并自动产出每个方法的"覆盖准则"。确保 P6 在构建场景时知道"要做到什么程度才算测够"。

## 环节性质：一次性（One-shot）

| 维度 | 说明 |
|------|------|
| **模式** | **一次性匹配决策**，非循环迭代 |
| **原因** | 输入（P4 测试点）已是稳定、穷举完成的产物。P5 只做"查表 + 匹配 + 轻量验证"，不做补充分析 |
| **错误处理** | 发现 High 优先级测试点无系统方法可配 → **回溯 P4**；发现模型缺少方法所需信息 → **回溯 P3** |

## 输入标准（不满足不启动）

P5 启动前，P4 输出必须满足：

| 检查项 | 要求 |
|--------|------|
| 测试点清单完整性 | 每个测试点包含：ID、描述、归属分类（#1-#9）、优先级（H/M/L） |
| 分类标签有效性 | 归属分类不得为 `Unclassified` 或空白 |
| 优先级标签有效性 | 优先级不得为空白 |

**硬性门禁**：若存在 `Unclassified` 测试点，P5 无法查表匹配，须立即回溯 P4 补齐分类。

## 核心映射规则（软推荐）

以下为 **"分类 → 设计方法"软推荐映射表**。测试专家可根据实际上下文调整，但需在矩阵中注明调整理由。

| P4 分类 | 主设计方法（推荐） | 补充方法（可选） | 方法适用场景说明 |
|--------|-------------------|----------------|----------------|
| #1 功能正确性 | 用例测试（Use Case Testing） | 决策表 | 验证操作与输出的匹配，天然适合用例场景 |
| #2 业务规则与逻辑 | 决策表测试（Decision Table Testing） | 因果图 | 涉及多条件组合规则，决策表可穷举组合 |
| #3 输入边界与等价类 | 等价类划分 + 边界值分析（ECP + BVA） | 组合测试（Pairwise） | 输入域分析的标准捆绑技术 |
| #4 状态转换 | 状态转换测试（State Transition Testing） | N/A | 有明确状态机时，此方法不可替代 |
| #5 异常与健壮性 | 错误猜测（Error Guessing） | 用例测试（异常路径） | 异常路径依赖经验，配合负面用例结构化 |
| #6 交互与集成 | 用例测试（Use Case Testing） | 接口测试（语义/时序） | 验证多对象协作时的数据流与调用时序 |
| #7 权限与安全 | 用例测试（不同 Actor 变体） | 决策表（权限矩阵） | 不同角色执行同一操作，本质是不同输入组合 |
| #8 非功能性 | 基于经验的测试（Experience-based） | 操作剖面测试 | ISTQB 对性能无标准黑盒技术，标注为经验法 |
| #9 数据完整性 | 决策表测试（事务/并发规则） | 状态转换测试 | 涉及事务一致性、并发冲突时，组合条件是关键 |

## 优先级决策规则（带风险意识）

P5 根据测试点优先级（H/M/L）决定方法选择的**强制性**：

| 优先级 | 方法选择策略 | 无标准系统方法时的处理 |
|--------|-------------|----------------------|
| **High** | **强制系统法**：必须匹配上表中的系统性方法（ECP/BVA/决策表/状态转换/用例测试）。不得以"经验法"作为主方法 | **强制回溯 P4**：重新审视该测试点是否粒度过粗或分类不当。不允许进入 P6 |
| **Medium** | **优先系统法**：优先匹配上表中的系统性方法。若模型特征不适配（如无状态机却选了状态转换），允许降级 | **允许经验法兜底**：但必须在矩阵"选择理由"列中标注"降级原因"，例如："因模型无明确状态机，改用错误猜测" |
| **Low** | **允许经验法**：可直接使用经验法（Error Guessing / Exploratory），无需强制匹配系统方法 | 直接采用经验法，无需回溯 |

## 覆盖准则自动映射表（由 P5 自动填充）

P5 选定主方法后，**自动带出**对应的覆盖准则（写入输出矩阵的"覆盖准则"列），无需测试专家人工填写：

| 主设计方法 | 自动带出的覆盖准则 |
|-----------|-------------------|
| 等价类划分 + 边界值分析（ECP+BVA） | 覆盖所有有效等价类（至少 1 个）、无效等价类（至少 1 个）；覆盖闭区间内点、边界值（min/max）、开区间外点（±1） |
| 决策表测试（Decision Table） | 覆盖所有条件组合（全组合）；若条件数 ≥4，建议使用 MCDC（修正条件判定覆盖） |
| 状态转换测试（State Transition） | 覆盖所有合法跃迁（Happy Path）；覆盖所有非法跃迁（Exception Path）；覆盖 0-1-多次循环 |
| 用例测试（Use Case Testing） | 覆盖主成功场景（Happy Path）；覆盖所有备选流（Alternate Path）；覆盖所有异常流（Exception Path） |
| 错误猜测（Error Guessing） | 基于测试专家经验，覆盖常见错误场景、边界溢出、空指针、超时等（无固定数学覆盖标准，输出场景清单） |
| 基于经验的测试（Experience-based） | 覆盖典型负载点（如并发数=1/峰值/均值）、典型数据量边界（基于历史故障模式） |
| 组合测试（Pairwise） | 覆盖所有参数对的取值组合（两两组合），若参数数 ≤3 则覆盖全组合 |

## 轻量验证机制（P5 的自我质量门禁）

P5 完成方法匹配后，必须执行轻量验证，检查**方法所需的前置模型信息是否在 P3 中存在**。结果写入输出矩阵"轻量验证结果"列。

| 选中的方法 | 验证逻辑（P3 模型必须包含什么） |
|-----------|------------------------------|
| 状态转换测试 | 检查 P3 模型中该测试点对应的对象是否**有明确的状态机定义**（状态列表 + 合法跃迁表）。无 → 标 **Fail**，回溯 P3 补充 |
| 决策表测试 | 检查 P3 模型中该测试点是否涉及**多条件组合规则**（如不变量/约束含 ≥2 个条件变量）。无条件组合 → 标 **Fail**，回溯 P4 重新审视分类 |
| 等价类/边界值 | 检查 P3 模型中该属性是否**有明确数据类型和值域**（如 int, 1-100）。无明确值域 → 标 **Fail**，回溯 P3 补充属性约束 |
| 用例测试 | 检查 P3 模型中该操作是否**有明确输入参数和输出定义**。无 → 标 **Fail**，回溯 P3 |
| 经验法（Error Guessing / Experience-based） | **自动 Pass**（经验法不依赖模型形式化定义，无需验证） |

> **验证结果处理**：若验证结果为 **Fail**，P5 不阻塞整条流水线，但必须在矩阵中标红告警，并建议**优先回溯 P3/P4 修复**。若涉及 High 优先级测试点且无法修复，则按"High 无标准方法"规则**强制阻塞**。

## 输出：测试设计策略矩阵

P5 完成后，必须交付以下矩阵，作为 **P6（测试场景构建）的唯一输入**：

| 字段名 | 必要性 | 来源/规则 |
|--------|--------|----------|
| 测试点 ID | 必须 | 来自 P4 |
| 测试点描述 | 必须 | 来自 P4 |
| 归属分类 | 必须 | 来自 P4（#1-#9） |
| 优先级 | 必须 | 来自 P4（H/M/L） |
| 主设计方法 | 必须 | 查"软推荐映射表"命中，若有调整需标注 |
| 补充设计方法 | 可选 | 如有 |
| **覆盖准则** | **必须** | 由 P5 按"覆盖准则自动映射表"填充 |
| 轻量验证结果 | 必须 | Pass / Fail（附 Fail 原因） |
| 选择理由 | 可选 | 若主方法与推荐表不一致，或 Medium 降级经验法，需填写理由 |

|---------|------|------|--------|-----------|---------|---------|---------|

## 完成标志与下一阶段

- **完成标志**：所有测试点均完成方法匹配 + 轻量验证扫描 + 矩阵输出。
- **存在 Fail**：输出矩阵中标注 Fail 的测试点，需人工决策是否回溯 P3/P4。若 High 优先级 Fail → **强制阻塞**，不允许进入 P6。
- **进入下一阶段**：矩阵确认无误后，进入 [测试场景构建](p6-test-scenario-construction.md)。
