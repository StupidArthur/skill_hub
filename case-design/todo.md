## 🔴 问题 1（高优先级）：P5 的“覆盖准则”与“优先级”存在隐性冲突

你在 P5 中定了两个规则：
- **规则 A**（决策点 2）：Low 优先级允许直接用经验法，无需强制系统法。
- **规则 B**（覆盖准则自动映射表）：决策表测试 → “若条件数≥4，建议使用 MCDC”。

**冲突场景**：一个 Low 优先级的业务规则测试点（#2），条件数为 5。P5 查表命中了“决策表”，然后自动带出覆盖准则“若条件数≥4，建议 MCDC”。但 Low 优先级已经允许用经验法了。

**结果**：P5 输出矩阵里会同时出现“主方法=经验法”和“覆盖准则=MCDC”——**自相矛盾**。

**建议修复**：
在 P5 覆盖准则自动映射表中加一行前置逻辑：
> **若该测试点优先级 = Low，且主方法为经验法，则不自动带出系统方法的覆盖准则，覆盖准则写为：“基于经验，覆盖典型故障场景”**。
> MCDC 建议仅对 High/Medium 优先级生效。

## 🔴 问题 2（高优先级）：P2 的“Dependency”到 P4 的“#6 交互”之间缺少强制链路

你在 P2 定义了 8 类元素，其中 **Dependency（依赖）** 可能是“无”（N/A），也可能是“依赖支付网关”、“依赖库存服务”等。

P4 的九大分类里有 **#6 交互与集成**，但 P4 的提取逻辑是“遍历 P3 模型要素”。如果 P3 在建模时把 Dependency 建模成了普通属性而非独立对象，P4 遍历时可能**漏掉**交互类测试点。

**风险**：一个外部依赖（如支付网关超时）没有被提取为 #6 测试点，导致 P6 的 Recovery Path 场景无法生成，最终漏测。

**建议修复**：
在 **P4 的输入标准**中加一条硬性门禁：
> **若 P2 输出的 Dependency 元素中有任何非 N/A 的依赖项，则 P4 必须确保在 #6 交互与集成 分类下至少有 1 个对应的测试点，否则视为“提取遗漏”，不允许进入 P5。**

## 🟡 问题 3（中优先级）：P1 的“确认不存在”在 P2 中可能被误判为“缺失”

你在 P1 的 G4 中规定：
> Constraint / Exception 必须“已明确或**确认不存在**”。

但 P2 是“机械解构”，它读到的可能是一个空白字段。如果 P1 的澄清报告中没有明确标注“该字段已确认不存在”，P2 看到空白会误以为是“缺失”，从而触发回溯 P1——**造成无效循环**。

**建议修复**：
在 P1 的输出格式中，强制要求：
> 对于“确认不存在”的元素，在 P2 的结构化元素卡片中写入 **`N/A (Confirmed)`**，而非留空或写 `无`。
> P2 的输入检查规则明确为：**只允许出现“有值”或“N/A (Confirmed)”，不允许出现空白或“无”**。

## 🟡 问题 4（中优先级）：P2 的“State”与 P3 的“状态机”之间缺少显式的“构建触发器”

P2 的 8 类元素里有 `State`，P3 的输出里有“状态机描述”。但 P2 的 State 是“该对象有哪些状态”（如 Draft/Pending/Paid），而 P3 的状态机需要的是“状态列表 + **合法跃迁表**”。

**风险**：P2 提取了状态列表，但 P3 建模时如果没有显式要求去读“状态跃迁规则（来自需求规格的业务流程）”，P3 可能只列了状态但没画跃迁边。P4 的 #4 状态转换测试点提取时发现状态机不完整 → 回溯 P3，**但 P3 会问“我没数据”又回溯 P2**，链条拉长。

**建议修复**：
在 **P3 的输出标准**中，对状态机的定义加一条硬性要求：
> **状态机必须包含“合法跃迁表”**（从哪个状态、通过哪个操作、转到哪个状态）。合法跃迁表的来源是 P2 的 Goal（操作）和 State 的组合，结合需求规格中的业务流程。
> 若 P2 有 State 但 P3 无法推导出合法跃迁（需求中没写），则 P3 阻塞并 **回溯 P1**，要求补充状态跃迁规则。

## 🟢 问题 5（低优先级/表达层面）：P4“九大分类”在 SKILL.md 中只列了名称，未内联定义

`SKILL.md` 中写的是“输出：测试点集合（按九大分类）”，但九大分类的具体内容散落在我们 P4 的讨论中，在 `p4-test-point-extraction.md` 正式稿里需要显式列出。

**影响**：如果测试专家只看 `SKILL.md`（顶层），不知道九大分类是哪九个，会疑惑。

**建议**：在 `SKILL.md` 的“横切规则”表格中，把九大分类的定义**直接内联**（至少列名称和一句话说明），或明确指向 `p4` 文档。这样顶层文件自成体系。

## 🟢 问题 6（低优先级/范围外）：缺少“执行反馈 → P1 需求修正”的闭环路径

当前流水线是 **P1→P7 单向**。实战中，测试执行（基于 P7 用例）发现缺陷，如果缺陷根因是需求缺失/错误，需要修正需求。

**现状**：流程在 P7 结束，没有定义“如何回到 P1”。

**建议**：
在 `SKILL.md` 末尾或 `p7` 文档中加一段说明（1-2 句话即可）：
> **反馈闭环**：若测试执行发现缺陷根因属于需求缺失或错误，需将问题反馈至 P1，重新启动需求澄清循环，修改后的需求再次进入流水线。本流水线为“可重入”设计，P1 支持多轮迭代，P2-P7 为一次性线性推进。
