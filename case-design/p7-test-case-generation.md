# P7 测试用例生成（Test Case Generation）

> 流程阶段⑦。承接 P6 的场景蓝图，将其机械翻译为标准格式的测试用例。P7 是流水线的最后一米，本质是"格式化输出器"——不再进行任何业务推理，只做精确翻译。

## 核心使命

把"故事"翻译成"剧本"。

P6 输出的是业务叙事（"用户创建订单"），P7 要把它翻译成测试执行人员可直接操作的精确指令（"在金额输入框输入'100'，点击'提交'按钮"）。所有数据嵌入步骤中，不单独列表，确保用例连贯可读。

## 环节性质：一次性（One-shot）

| 维度 | 说明 |
|------|------|
| **模式** | **一次性机械翻译**，非循环迭代 |
| **原因** | 输入（P6 场景蓝图）已是完整、稳定的业务编排产物。P7 只做"格式转换+执行方式填充" |
| **错误处理** | 发现 P6 场景缺少具体数据值或步骤歧义 → **阻塞**，回溯 P6 补充，不允许 P7 自行臆测 |

## 输入标准（不满足不启动）

| 输入来源 | 要求 |
|---------|------|
| P6 场景蓝图集合 | 每个场景含：场景 ID、步骤序列（业务叙事）、代表性数据、预期业务结果、关联测试点 ID |
| P3 领域模型 | 用于确认数据的有效性（如值域范围） |
| 操作手册 / UI 原型 / 接口文档 | 用于确定具体执行方式（默认以 UI 操作为主，若涉及 API 需显式说明） |
| **硬性门禁** | P6 场景中不得有"未指定代表性数据"的步骤。若有，P7 无法翻译，须回溯 P6 补充 |

## 输出标准（P7 完成后必须交付）

一份标准测试用例集合，每个用例包含以下字段：

| 字段 | 含义 | 示例 |
|------|------|------|
| **用例 ID** | 唯一编号，可回溯 | TC-001 |
| **用例标题** | 简短业务描述 | "有效订单提交并支付成功" |
| **关联场景 ID** | 回溯到 P6 | SC-001 |
| **关联测试点 ID** | 回溯到 P4/P5 | TP-001, TP-015 |
| **前置条件** | 执行前必须满足的状态/数据准备 | 用户 test_user 已登录；购物车有商品 A（单价 100） |
| **测试步骤** | 精确操作指令（数据嵌入步骤中） | 见下方规范 |
| **预期结果** | 可验证的断言（含界面反馈、状态变化、数据变更） | 见下方规范 |
| **备注（可选）** | 特殊注意事项（如"需提前准备测试账号"） | — |

## 测试步骤编写规范（数据嵌入）

P7 的测试步骤必须把数据**直接嵌入**操作描述中，不单独列"输入数据"字段，以保证用例在阅读时一气呵成：

| ✅ 正确写法（数据嵌入步骤） | ❌ 错误写法（数据脱离步骤） |
|---------------------------|--------------------------|
| 1. 在"金额"输入框输入 `100`<br>2. 点击"提交"按钮 | **输入数据**：金额=100<br>**步骤 1**：输入金额<br>**步骤 2**：点击提交 |

## 默认执行方式

- **当前流水线默认以 Web UI 操作为主**。所有操作步骤按"在 XX 页面，点击 XX 按钮，在 XX 输入框输入 XX 值"的格式编写。
- 若涉及 **API/接口调用**，需在用例"备注"中显式说明（如"本用例通过 API 执行"），并在步骤中按 API 格式描述（如"调用 POST /api/order/submit"）。

## 复杂 UI 输出的校验规范（关键补充）

当 P6 场景的"预期业务结果"涉及**非结构化 UI 输出**（如图表、文件导出、可视化大屏）时，P7 必须将校验点细化到**可操作、可断言**的层面，不能只写"验证图表正确"或"验证文件导出成功"。

### 场景1：UI 输出为"趋势图/图表"

| P6 写的预期（业务层面） | P7 必须翻译成的校验描述（可执行层面） |
|-----------------------|-------------------------------------|
| "系统展示温度趋势图" | **校验点**：<br>1. 页面渲染出折线图（canvas/svg 元素存在）<br>2. X 轴显示时间范围 `13:00 ~ 14:00`<br>3. Y 轴显示温度范围 `0°C ~ 100°C`<br>4. 数据点在 `13:30` 处值为 `75°C`（峰值）<br>5. 图例显示"温度"曲线 |

**原则**：图表校验 = **坐标轴范围 + 数据点特征 + 视觉元素存在性**。P7 需从 P3 模型中读取"温度值域[0,100]"和"时间采样点"来填充具体校验值。

### 场景2：UI 输出为"导出文件（CSV/PDF/Excel）"

| P6 写的预期（业务层面） | P7 必须翻译成的校验描述（可执行层面） |
|-----------------------|-------------------------------------|
| "系统导出订单报表" | **校验点**：<br>1. 浏览器触发下载，文件名为 `order_report_YYYYMMDD.csv`<br>2. 文件格式为 CSV，编码 UTF-8<br>3. 表头包含：`订单号, 金额, 状态, 创建时间`<br>4. 数据行数 = 当日订单总数（≥1）<br>5. 首行数据与界面列表首条记录一致（金额/状态可比对） |

**原则**：文件校验 = **文件名/格式/表头/行数/抽样内容比对**。P7 需明确校验手段（如"打开 CSV 验证"），不能只写"验证文件正确"。

### 场景3：UI 输出为"提示信息/弹窗/状态灯"

这类输出相对简单，P7 按标准断言描述即可：

| P6 写的预期 | P7 翻译 |
|-----------|--------|
| "系统提示提交成功" | 页面顶部显示绿色提示框，内容包含"提交成功" |
| "系统拒绝并提示错误" | 弹窗显示红色错误提示，内容包含"金额必须大于 0" |
| "状态灯变为绿色" | 订单状态标签变为绿色，文字显示"已支付" |

## 完整用例示例

### 示例1：标准 UI 操作（含图表校验）

**TC-001：查看温度趋势图**
- **关联场景**：SC-002
- **关联测试点**：TP-008, TP-012
- **前置条件**：温度传感器已连接，历史数据已生成（时间范围 13:00-14:00）
- **测试步骤**：
  1. 登录系统，进入"温度监控"大屏
  2. 在"时间范围"选择器中选择 `13:00 ~ 14:00`
  3. 点击"查询"按钮
- **预期结果**：
  - 页面渲染出折线趋势图（canvas 元素存在）
  - X 轴显示时间范围 `13:00 ~ 14:00`，间隔 30 分钟
  - Y 轴显示温度范围 `0°C ~ 100°C`
  - 数据点在 `13:30` 处显示峰值 `75°C`
  - 图例显示"温度"曲线
- **备注**：无

### 示例2：含文件导出校验

**TC-002：导出订单报表**
- **关联场景**：SC-005
- **关联测试点**：TP-020
- **前置条件**：当前账号有 3 条已完成订单（状态=Paid）
- **测试步骤**：
  1. 登录系统，进入"订单管理"页面
  2. 点击"导出报表"按钮
  3. 等待浏览器完成下载
- **预期结果**：
  - 浏览器触发下载，文件名匹配 `order_report_*.csv` 格式
  - 打开 CSV 文件，表头包含：`订单号, 金额, 状态, 创建时间`
  - 数据行数 = 3（与前置条件一致）
  - 首行订单金额与界面列表首条记录一致
- **备注**：需提前准备 CSV 读取工具（如 Excel 或文本编辑器）

### 示例3：标准 UI 操作（无特殊输出）

**TC-003：有效订单提交成功**
- **关联场景**：SC-001
- **关联测试点**：TP-001, TP-015
- **前置条件**：用户 test_user 已登录；购物车有商品"商品 A"（单价 100 元）
- **测试步骤**：
  1. 进入"订单管理"页面
  2. 点击"新建订单"按钮
  3. 在"商品名称"输入框输入 `商品A`
  4. 在"数量"输入框输入 `1`
  5. 在"金额"输入框输入 `100`
  6. 点击"保存"按钮，生成订单草稿
  7. 在订单详情页，点击"提交"按钮
- **预期结果**：
  - 订单状态变为"待支付（Pending）"
  - 页面顶部显示绿色提示"提交成功"
  - 库存中"商品 A"数量减少 1
- **备注**：无

## P7 自我约束（防止越界）

| ✅ P7 应该做的 | ❌ P7 不应该做的 |
|-------------|---------------|
| 把 P6 的业务叙事翻译成精确的 UI 操作指令 | 新增 P6 中没有的测试步骤或验证点 |
| 从 P3 值域中查表填充 P6 未指定具体值的数据 | 凭空发明新数据类别或新业务场景 |
| 参考操作手册/UI 原型确定具体操作方式 | 在用例中讨论"该不该测这个" |
| 按 IEEE 829 标准格式化输出 | 修改 P6 的步骤顺序或业务逻辑 |
| 对图表/文件输出，将校验细化到坐标轴/表头级别 | 只写"验证图表正确"这种不可执行描述 |
| 每条用例标注关联的场景 ID 和测试点 ID | 产出无法回溯到上游的孤立用例 |

## 完成标志与流程结束

- **完成标志**：所有 P6 场景均已翻译为至少 1 条标准测试用例；每条用例均标注关联场景 ID 和测试点 ID，可完整回溯至 P4/P3/P1。
- **存在未翻译场景**：若有场景因"缺少操作手册"或"UI 交互不明确"无法翻译 → **阻塞**，需人工补充输入。
- **流程结束**：P7 输出完成 → **整条流水线（P1→P7）结束**，无下一阶段。产出物交付测试执行团队。
